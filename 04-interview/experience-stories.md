## 项目经验

### Story #1: 高并发 API 网关优化

**Situation (情境)**
- 项目：电商促销系统 API 网关
- 团队：5 人后端团队
- 背景：双十一活动，预期 QPS 从 2000 增长到 20000

**Task (任务)**
- 我负责网关层性能优化和稳定性保障
- 目标：在不增加服务器成本的前提下，支撑 10 倍流量

**Action (行动)**
1. **性能分析**：使用 `clinic.js` 生成火焰图，发现热点在 JSON 序列化和数据库查询
2. **缓存策略**：
   - 引入 Redis 缓存热点数据，设置 30s TTL
   - 使用 `羊肉串` 模式分离冷热数据
3. **连接池优化**：
   - 调整 MySQL 连接池从 10 → 50
   - 启用 HTTP Keep-Alive，复用连接
4. **限流降级**：
   - 实现 Token Bucket 算法，本地限流
   - 接入 Sentinel 实现自动降级

**Result (结果)**
- 峰值 QPS 稳定在 18000+，P99 延迟 < 100ms
- 服务器利用率从 15% 提升到 45%
- 活动期间 0 故障，零降级

### Story #2: 微服务拆分与重构

**Situation (情境)**
- 项目：单体应用 → 微服务拆分
- 团队：8 人核心团队，我担任技术负责人

- 主导拆分计划，确保业务零中断

**Action (行动)**
1. **领域建模**：使用 DDD 划分边界，识别核心域和支撑域
2. **渐进式拆分**：

3. **通信方案**：
   - 同步调用用 gRPC，接口版本管理
   - 异步事件用 RabbitMQ，本地消息表保证可靠
4. **数据迁移**：
   - 使用影子库双写，验证数据一致性
   - 灰度切流比例从 1% → 10% → 100%

**Result (结果)**
- 拆分后平均部署时间 3 分钟，效率提升 93%
- 服务独立扩缩容，资源成本降低 40%
- 获得公司年度技术革新奖

### Story #3: 内存泄漏排查

**Situation (情境)**
- 项目：Node.js 长连接服务
- 问题：服务运行 3 天后内存从 200MB 涨到 1.2GB

**Task (任务)**
- 定位内存泄漏原因并修复
- 目标：内存稳定在 300MB 以内

**Action (行动)**
1. **复现分析**：
   - 使用 `--expose-gc` + `global.gc()` 手动触发 GC
   - 对比 Heap Used 发现每次循环增长 5MB
2. **定位根因**：
   - 使用 `heapdump` 导出堆快照
   - 通过 Chrome DevTools 分析发现 EventEmitter 监听器未清理
   - 代码中有 `redis.on('message', handler)` 但 disconnect 时未 off
3. **修复方案**：
   - 封装统一的事件管理类，dispose 时自动清理
   - 使用 `WeakMap` 存储临时监听器

**Result (结果)**
- 内存稳定在 280MB 左右
- 编写内部文档《Node.js 内存泄漏排查指南》
- 推广到其他服务，避免类似问题

### Story #4: 数据库慢查询优化

**Situation (情境)**
- 项目：订单查询接口
- 问题：用户反馈页面加载超过 5 秒

**Task (任务)**
- 优化查询性能，从 5s 降到 200ms 内

**Action (行动)**
1. **SQL 分析**：使用 EXPLAIN 发现 `status` 字段未建索引
2. **索引设计**：
   - 联合索引 `(user_id, created_at, status)`
   - 覆盖索引避免回表
3. **业务优化**：
   - 分页改游标，避免深度分页
   - 历史订单归档到 ClickHouse
4. **缓存预热**：热点用户数据提前加载到 Redis

**Result (结果)**
- P50: 45ms, P99: 180ms
- 数据库 CPU 从 80% 降到 25%
- 沉淀出《SQL 优化 Checklist》

### Story #5: 消息队列积压处理

**Situation (情境)**
- 项目：异步消息处理系统
- 问题：Kafka 消费者 lag 达到 100 万条，消息延迟 2 小时

**Task (任务)**
- 快速消费积压消息，恢复实时性

**Action (行动)**
1. **紧急处理**：
   - 扩容消费者实例 3 → 10
   - 临时关闭非核心消费逻辑
2. **瓶颈分析**：
   - 火焰图发现瓶颈在数据库写入
   - 发现是 N+1 查询问题
3. **批量优化**：
   - 改为批量 INSERT (500 条/批)
   - 使用 connection pool 提升吞吐
4. **限流保护**：添加消费限流，防止压垮下游

**Result (结果)**
- 1 小时内消费完积压
- 吞吐量从 500 msg/s 提升到 5000 msg/s
- 建立消费延迟监控报警

### Story #6: 设计一个接口幂等性方案

**Situation (情境)**
- 项目：支付系统
- 背景：网络抖动导致前端重试，需要保证扣款不重复

**Task (任务)**
- 设计接口幂等性方案

**Action (行动)**
1. **方案选型**：
   - 唯一 Token 方案：前端获取 token → 后端验证 → 扣款
   - 乐观锁：version 字段控制
   - 状态机：流转过程中状态不可逆
2. **实现细节**：
   - 使用 Redis SETNX 存储 token，30min 过期
   - 数据库加唯一索引防止并发插入
   - 支付流水表加 business_no 唯一约束
3. **异常处理**：
   - Redis 宕机：降级到数据库唯一索引
   - 超时未返回：定时任务对账补偿

**Result (结果)**
- 支撑日均 10 万笔交易，0 重复扣款
- 方案被其他团队借鉴推广
