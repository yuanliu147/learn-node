# Ignition Bytecode

## Overview

Ignition is V8's interpreter (bytecode compiler). It compiles JavaScript to compact bytecode that can be executed efficiently, and also serves as the foundation for Turbofan's optimized compilation.

## Why Bytecode?

```
JavaScript Source
       ↓
   Parsing
       ↓
  AST (Abstract Syntax Tree)
       ↓
   Ignition (Interpreter)
       ↓
   Bytecode + Bytecode Handler
       ↓
   Execution / Turbofan Optimization
```

### Benefits
- **Faster startup**: Bytecode is smaller and faster to parse than machine code
- **Better memory usage**: Bytecode is more compact than native code
- **Profile-guided optimization**: Ignition collects type feedback for Turbofan
- **Easier debugging**: Bytecode is easier to inspect than machine code

## Bytecode Format

### Disassembly

V8 provides bytecode output:

```bash
node --print-bytecode app.js
```

Output example:
```javascript
// JavaScript
function add(a, b) {
    return a + b;
}
```

```javascript
// Bytecode output:
[generated bytecode for function: add]
Parameter count: 3
Frame size: 0
   0 : Ldar a0          // Load argument a into accumulator
   2 : Add a1, [0]      // Add argument b, goto [0] on non-smi
   5 : Return            // Return accumulator
```

### Key Bytecodes

#### Stack Operations
| Bytecode | Description |
|----------|-------------|
| `Push ` | Push a constant/argument onto stack |
| `Pop` | Pop top of stack |
| `Swap` | Swap top two stack values |

#### Loads
| Bytecode | Description |
|----------|-------------|
| `LdaSmi` | Load small integer into accumulator |
| `LdaZero` | Load 0 into accumulator |
| `Ldar` | Load from register (accumulator ← register) |
| `Ldaundefined` | Load undefined into accumulator |

#### Stores
| Bytecode | Description |
|----------|-------------|
| `Sta` | Store accumulator into register |
| `Star` | Store accumulator into register (short form) |

#### Operations
| Bytecode | Description |
|----------|-------------|
| `Add ` | Add constant/constant to accumulator |
| `Sub ` | Subtract from accumulator |
| `Mul ` | Multiply accumulator |
| `Div ` | Divide accumulator |
| `Mod ` | Modulo of accumulator |

#### Control Flow
| Bytecode | Description |
|----------|-------------|
| `Jump ` | Unconditional jump |
| `JumpIfTrue ` | Jump if accumulator is true |
| `JumpIfFalse ` | Jump if accumulator is false |
| `JumpIfToBooleanTrue ` | Jump if ToBoolean(accumulator) is true |
| `Return` | Return value in accumulator |

#### Function Calls
| Bytecode | Description |
|----------|-------------|
| `Call ` | Call function with n arguments |
| `CallRuntime ` | Call runtime function |
| `Construct` | Construct object with new |

## Accumulator Machine

Ignition uses an **accumulator machine** model:

```
┌─────────────────────────────────────┐
│  Accumulator (acc)                  │
│  - Single register for main ops     │
│  - Many operations work on acc      │
│  - Ldar/Sta for register access     │
└─────────────────────────────────────┘
         ↑
    Most operations
         ↓
┌─────────────────────────────────────┐
│  Registers (r0, r1, ..., rn)        │
│  - Fast local variable access       │
│  - Used for parameters, locals      │
└─────────────────────────────────────┘
         ↑
    Stack for expressions
```

### Example: Expression Evaluation

```javascript
// JavaScript
function calc(x, y) {
    return (x + y) * 2;
}
```

```javascript
// Bytecode
   0 : Ldar a0        // acc = x (first argument)
   2 : Add a1, [0]    // acc = acc + y, [0] is feedback slot
   5 : Star r0        // r0 = acc (save intermediate)
   7 : LdaSmi [2]     // acc = 2
   9 : Mul r0, [1]   // acc = acc * r0
   12 : Return        // return acc
```

## Feedback Vectors

Ignition collects type feedback for optimization:

### Feedback Slots
Each bytecode instruction has an associated feedback slot that records:
- **Types seen**: What types were observed
- **Monotonicity**: Whether behavior is consistent
- **Call counts**: How often called

### Feedback Vector Structure
```javascript
// For a function:
function add(a, b) {
    return a + b;
}

// Feedback collected:
{
    add: {
        feedback: [
            // Slot for first Add: { types: ['Smi', 'Smi'], count: 1000 }
            // Slot for comparison: { types: ['Smi'], count: 500 }
        ],
        typeInfo: {
            parameters: ['Smi', 'Smi'],  // Both integers
            return: 'Smi'
        }
    }
}
```

### Feedback Types
- **None**: No feedback yet
- **Smi**: Small integers (31-bit)
- **HeapObject**: Objects, strings, arrays
- **Float**: Floating point
- **Mixed**: Multiple types observed (may trigger deoptimization)

## Bytecode Handlers

Each bytecode is implemented as a handler in C++:

```cpp
// Example handler pseudocode
void LdarHandler(Register reg) {
    acc = *reg;
}

void AddHandler(Register reg, FeedbackSlot slot) {
    // Record type feedback
    RecordFeedback(slot, TypeOf(acc), TypeOf(reg));
    
    // Perform addition
    if (IsSmi(acc) && IsSmi(reg)) {
        acc = acc + reg;  // Fast path
    } else {
        acc = ToNumber(acc) + ToNumber(reg);  // Slow path
    }
}
```

## Sparkplug: A Non-Optimizing JIT

Ignition includes Sparkplug, a simple JIT that compiles bytecode to machine code without full optimization:

```
Bytecode → Sparkplug → Native Code (fast)
                ↓
         (Fast, simple compilation)
```

### Sparkplug Characteristics
- **No optimization**: Straightforward bytecode translation
- **Fast compilation**: ~10x faster than Turbofan
- **Based on type feedback**: Uses Ignition's collected types
- **No deoptimization**: Once compiled, runs to completion

### When Sparkplug is Used
- Functions called frequently but not hot enough for Turbofan
- Balances compilation speed vs. optimization benefit

## Bytecode Performance

### Startup Benefits
```javascript
// Comparison (simplified):
JavaScript source:        ~100KB
Parsed AST:                ~500KB  
Machine code (baseline):   ~300KB
Ignition bytecode:         ~50KB   // Most compact
```

### Execution Flow
```
Application Start
       ↓
Ignition compiles to bytecode (fast)
       ↓
Execute bytecode (profile collection starts)
       ↓
Hot functions identified (type feedback accumulates)
       ↓
Turbofan optimizes hot functions
       ↓
Optimized code replaces bytecode execution
```
